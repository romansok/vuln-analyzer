# CWE-200: Information Exposure

## Plain-English explanation
The app reveals data it shouldn't — internal IDs, file paths, stack traces, hidden user fields, secrets in logs. Each leak is small on its own; chained together they tell an attacker enough to find the next bug.

## Typical attack pattern
A user-profile API returns the full DB row, including `password_hash`, `email_verification_token`, and `internal_admin_notes`. The frontend only displays `name` and `avatar`, but the JSON over the wire has everything. An attacker browses the network tab and reads other users' tokens.

## Reachability hints
- Endpoints that return whole DB rows / ORM model `.to_dict()` / `JSON.stringify(row)`.
- Error handlers that send stack traces back to the client (especially with `NODE_ENV !== 'production'`).
- Detailed error messages: "User not found" vs "Wrong password" — enumeration.
- Verbose logs that include request bodies (passwords, tokens, PII).
- 404 vs 403 distinctions revealing which IDs exist.

## Common workarounds when a bump is blocked
- Allowlist serializer at the response boundary — define exactly which fields go out, never `JSON.stringify(model)`.
- Generic error responses: `{ error: "invalid request" }` in prod; details only in server logs.
- Scrub the logger: redact `password`, `token`, `secret`, `authorization` keys at the log middleware.
- Unify response codes for enumeration-prone endpoints (login, password reset).

## Concrete code example

```js
// Vulnerable
app.get('/me', (req, res) => res.json(req.user));   // every field, including secrets

// Safe
app.get('/me', (req, res) => {
  const { id, name, avatar, createdAt } = req.user;
  res.json({ id, name, avatar, createdAt });
});
```
