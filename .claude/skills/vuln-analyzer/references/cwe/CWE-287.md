# CWE-287: Improper Authentication

## Plain-English explanation
The app's identity checks are missing, weak, or bypassable. The classic shapes: trusting a header the client sets (`X-User-Id`), accepting signed tokens without verifying the signature, accepting an `alg: none` JWT, or rolling a custom session token with too few bits of entropy.

## Typical attack pattern
A JWT library has a bug where it accepts a token with `alg: none` (no signature) and trusts the payload. An attacker crafts a JWT with `{"sub":"admin","alg":"none"}`, drops the signature segment, sends it as their auth token, and is now logged in as `admin`.

## Reachability hints
- `jwt.verify(token, key, { algorithms: [...] })` — missing `algorithms` option, or including `'none'`.
- `jwt.decode(token)` (no verification!) used in place of `verify`.
- Custom auth middleware that reads a user-id header and trusts it.
- Token comparison with `==` / `===` instead of constant-time comparison.
- Password hashing with bare MD5 / SHA1 / fast hashes with no salt.

## Common workarounds when a bump is blocked
- Always pin the `algorithms` allowlist on JWT verification. Banish `none`.
- Centralize auth in a single middleware; ban ad-hoc identity-from-header reads.
- Use `crypto.timingSafeEqual` (Node) / `hmac.compare_digest` (Python) for token comparisons.
- For password hashes: argon2 / bcrypt / scrypt with proper cost factors; never bare hash + salt.

## Concrete code example

```js
// Vulnerable
const claims = jwt.decode(token);              // no signature check!
req.user = await User.findById(claims.sub);

// Safe
const claims = jwt.verify(token, SECRET, { algorithms: ['HS256'] });
req.user = await User.findById(claims.sub);
```
