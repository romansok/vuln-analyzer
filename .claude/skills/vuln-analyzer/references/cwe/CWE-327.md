# CWE-327: Use of a Broken or Risky Cryptographic Algorithm

## Plain-English explanation
The code uses crypto that doesn't actually protect what it should — weak algorithm (MD5, SHA1, DES, RC4), weak mode (ECB), reused IV/nonce, predictable RNG, short key, or correct algorithm used in a way that defeats its guarantees.

## Typical attack pattern
A service signs API requests with `HMAC-MD5`. The attacker collects a few signed requests, runs a length-extension attack (HMAC-MD5 isn't actually vulnerable to length extension — but raw MD5(secret || msg) is; teams confuse them), and forges a request that grants admin.

## Reachability hints
- `createHash('md5'|'sha1')`, `createCipheriv('aes-128-ecb', ...)`, `crypto.DES`, `crypto.RC4`.
- `Math.random()` used to generate tokens, IDs, salts, IVs.
- Hard-coded IVs or salts (look for `Buffer.alloc(16, 0)` next to a cipher init).
- JWT with `HS256` and a short secret string ("changeme", "secret").
- Custom "encryption" — XOR with a key, base64 marketed as encrypted.

## Common workarounds when a bump is blocked
- Move hashes to SHA-256 / SHA-3 / BLAKE2; HMAC the result, don't roll your own MAC.
- Symmetric encryption: AES-GCM with a unique 12-byte IV per message; never reuse.
- RNG: `crypto.randomBytes(n)` / `secrets.token_bytes(n)` for anything user-facing.
- Replace JWT shared-secret with asymmetric (`RS256` / `EdDSA`) and a proper key.
- If the library itself is broken, isolate uses behind a thin wrapper module so swapping later is one change.

## Concrete code example

```js
// Vulnerable
const sig = crypto.createHmac('md5', SECRET).update(body).digest('hex');

// Safe
const sig = crypto.createHmac('sha256', SECRET).update(body).digest('hex');
// constant-time comparison on receipt
if (!crypto.timingSafeEqual(Buffer.from(sig), Buffer.from(provided))) reject();
```
