# CWE-862: Missing Authorization

## Plain-English explanation
The endpoint authenticates the caller — proves they are who they say — but never checks whether they're *allowed* to do the action. An attacker who's a regular user calls an admin endpoint, or reads another user's resource by guessing/incrementing an ID. Authentication ≠ authorization.

## Typical attack pattern
A REST API has `GET /api/orders/<id>` that returns the order. The handler verifies the JWT (you're logged in) but doesn't check the order belongs to you. The attacker iterates `id=1..1000000` and downloads every order in the system — classic "IDOR".

## Reachability hints
- Routes that take an ID directly from the URL and fetch from DB without a `WHERE user_id = ?` clause.
- Admin routes guarded only by "is logged in" middleware, not "has admin role".
- ORM-level checks missing (`findById` without scoping).
- `req.user.role === 'admin'` checked in one place but not the other.
- Indirect actions: file downloads, signed-URL generation, webhook subscriptions — all classic IDOR spots.

## Common workarounds when a bump is blocked
- Centralize a per-action authorization layer. Express middleware or a policy module that maps `(user, action, resource)` → allow/deny.
- Make the data layer enforce ownership by default: `findByIdForUser(id, userId)` is the only API; the bare `findById` is removed.
- Use opaque, unguessable IDs (UUIDv4, ULID) so iteration is at least expensive — but never *rely* on this; check authorization properly.
- Add integration tests asserting that user A can't read user B's resource for every resource type.

## Concrete code example

```js
// Vulnerable
app.get('/orders/:id', auth, async (req, res) => {
  const order = await Order.findById(req.params.id);   // any logged-in user, any order
  res.json(order);
});

// Safe
app.get('/orders/:id', auth, async (req, res) => {
  const order = await Order.findOne({ id: req.params.id, userId: req.user.id });
  if (!order) return res.status(404).end();
  res.json(order);
});
```
