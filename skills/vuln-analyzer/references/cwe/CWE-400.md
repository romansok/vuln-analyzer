# CWE-400: Uncontrolled Resource Consumption (DoS)

## Plain-English explanation
A small, cheap-to-send input causes large, expensive work on the server: pinning CPU, allocating gigabytes of memory, holding open file handles or sockets, queueing a billion timers. One attacker with a script knocks the service over.

## Typical attack pattern
A markdown renderer expands link references in O(n²) over the document size. The attacker sends a 1MB doc that triggers a tight loop running for minutes per request. They send a handful and the worker pool is exhausted; legitimate users get 503s.

## Reachability hints
- Parsers / regex engines run on user input without size limits.
- Compression (`gzip`, `zlib.inflate`) on untrusted input — zip-bombs.
- Image processing (`sharp`, `imagemagick`) on uploaded files without dimension limits.
- Recursive JSON / XML parsing — billion laughs, deeply nested.
- Database queries with user-controlled `LIMIT` / no `LIMIT` at all.
- Unbounded background work queued from a request handler.

## Common workarounds when a bump is blocked
- Body size limits at the edge: `express.json({ limit: '64kb' })`, nginx `client_max_body_size`.
- Timeouts everywhere: request timeout, regex timeout (`re2` instead of stdlib regex), DB query timeout.
- Bound parser inputs: image dimension caps, archive entry count caps, JSON depth caps.
- Per-IP rate limits and per-endpoint concurrency limits (semaphores, queues).

## Concrete code example

```js
// Vulnerable — regex with catastrophic backtracking on a long input
app.post('/match', (req, res) => {
  // (a+)+$ on "aaaa...aaab" runs for billions of steps
  res.json({ ok: /(a+)+$/.test(req.body.s) });
});

// Safer — bounded input, linear regex engine
const re2 = require('re2');
const RE = new re2('(a+)+$');
app.post('/match', (req, res) => {
  if (typeof req.body.s !== 'string' || req.body.s.length > 1024) return res.status(400).end();
  res.json({ ok: RE.test(req.body.s) });
});
```
